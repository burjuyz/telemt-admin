# Backlog дальнейшей реализации

Этот документ фиксирует следующий слой задач после базового перехода на `telemt` control API. Он нужен как рабочий список для дальнейших итераций и синхронизации документации, кода и эксплуатации.

## Приоритет 1. Завершить стабилизацию API-first backend

- Довести ручной rollout-checklist до реально повторяемой процедуры на staging и production.
- Зафиксировать expected error mapping для control API: какие ошибки показываются администратору, какие только логируются, а какие переводят сценарий на fallback.
- Добавить более явное логирование причин деградации в `telemt_backend`, чтобы было видно, когда сработал fallback и почему.
- Уточнить политику retry для mutation-запросов: где достаточно одного retry по `revision_conflict`, а где лучше останавливать сценарий.
- Отдельно проверить сценарии idempotency для create/delete, чтобы повторные админские действия не портили состояние.

## Приоритет 2. Довести модель данных и синхронизации

- Определить, когда `telemt` API считается окончательным источником истины для пользователя, а когда локальная БД ещё может выступать временным cache.
- Явно описать жизненный цикл полей `backend_mode`, `last_sync_error`, `last_seen_revision`, `last_synced_at`.
- Решить, нужно ли хранить дополнительный sync-state по последнему успешному API health-check или backend availability.
- Подготовить вторую волну миграций для перевода `secret` в legacy-only поле после стабилизации rollout.
- Продумать фоновую reconciliation-задачу: периодически сверять локальные approved-записи с `telemt` API и помечать рассинхрон.

## Приоритет 3. Улучшить UX и диагностику в Telegram

- Расширить `/service`, чтобы различались:
  - состояние systemd-процесса;
  - состояние control API;
  - состояние fallback-path;
  - состояние наблюдаемости и метрик.
- Добавить явное отображение backend-mode для конкретного пользователя или операции в админском UI.
- Продумать компактный diagnostic-screen для последних sync errors без чтения логов с сервера.
- Добавить человекочитаемые причины отказов по API: auth mismatch, timeout, whitelist block, revision conflict, not found.
- Решить, нужен ли отдельный админский action для принудительной sync-проверки пользователя через API.

## Приоритет 4. Встроить поддержку метрик и наблюдаемости

Сервер `telemt` уже отдаёт `/metrics`, поэтому `telemt-admin` должен уметь учитывать этот канал отдельно от control API.

- Определить модель использования `/metrics`:
  - только внешний Prometheus/VictoriaMetrics scrape;
  - scrape + краткая сводка в Telegram;
  - scrape + алерты + ссылка на внешний dashboard.
- Добавить конфиг observability-секции, например `[telemt_metrics]`, с параметрами:
  - `enabled`;
  - `base_url` или `metrics_url`;
  - `timeout_ms`;
  - `allow_insecure_localhost_only` или другой явный security-флаг, если это потребуется;
  - `dashboard_url`, если хотим показывать переход к внешнему dashboard.
- Реализовать лёгкий metrics-adapter для чтения и разбора Prometheus exposition format только по ограниченному набору метрик, нужных админке.
- Зафиксировать минимальный набор метрик, которые мы считаем operationally важными:
  - доступность `/metrics`;
  - общее состояние runtime/startup;
  - количество активных подключений или сессий, если такая метрика есть;
  - ошибки рукопожатий, reject rate, saturation-индикаторы, если они есть;
  - служебные counters по API или runtime events, если они коррелируют с проблемами доступа.
- Добавить в `/service` отдельный блок `Метрики`, который показывает:
  - свежесть scrape;
  - ключевые gauge/counter значения;
  - признак, что metrics недоступны, даже если control API ещё отвечает.
- Продумать правила деградации:
  - control API работает, а `/metrics` недоступен;
  - `/metrics` работает, а control API недоступен;
  - оба канала недоступны.
- Подготовить runbook для чтения метрик:
  - какие значения считать нормой;
  - какие значения считать поводом для ручной проверки;
  - какие сигналы использовать для автоматических алертов.
- Если в инфраструктуре уже есть Prometheus, задокументировать пример `scrape_config` и рекомендуемые alert rules.

## Приоритет 5. Сформировать полноценную операционную модель

- Описать рекомендуемую production-схему: `telemt-admin`, `telemt`, control API, `/metrics`, systemd, Prometheus, dashboard.
- Добавить в runbook раздел "симптом -> что проверить" для типовых проблем:
  - approve не проходит;
  - `/link` строится только через fallback;
  - control API отвечает, но runtime unhealthy;
  - `/metrics` недоступен;
  - данные UI расходятся с фактическим состоянием proxy.
- Зафиксировать политику ротации `auth_header` и действий после её изменения.
- Описать, как безопасно отключать fallback после стабилизации окружения.

## Приоритет 6. Упростить кодовую базу после стабилизации

- После подтверждения стабильности API-path уменьшить прямую зависимость от `telemt.toml` там, где она больше не нужна.
- Рассмотреть вынос HTTP-клиента control API и parser-а метрик в отдельные модули внутри backend-слоя.
- Сократить количество мест, где bot UI знает о технических деталях sync и fallback.
- Пересмотреть структуру документации после завершения rollout, чтобы удалить временные переходные оговорки.

## Предлагаемый порядок исполнения

1. Стабилизация backend и sync-модели.
2. Явная операционная диагностика в UI.
3. Поддержка `/metrics` и observability-конфиг.
4. Runbook и alerting вокруг метрик.
5. Упрощение legacy-зависимостей после подтверждения стабильности.
