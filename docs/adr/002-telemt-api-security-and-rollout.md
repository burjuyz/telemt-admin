# ADR 002: безопасность control API и стратегия rollout

## Статус

Принято

## Контекст

Control API `telemt` даёт прямой административный доступ к управлению пользователями и runtime-состоянию прокси. Это делает его удобной интеграционной точкой для `telemt-admin`, но одновременно повышает требования к безопасности и эксплуатации.

Во время анализа `../telemt` были подтверждены важные свойства:

- `[server.api]` не должен считаться hot-only: изменения обычно требуют restart `telemt`;
- у API есть отдельные механизмы security-gating: bind, whitelist, `Authorization`;
- endpoint `POST /v1/users/{username}/rotate-secret` в текущей серверной реализации фактически недостижим;
- `/metrics` и control API решают разные задачи и не заменяют друг друга.

## Решение

Для production-окружения принимаются следующие правила:

- control API должен слушать `127.0.0.1:9091` или приватный интерфейс;
- API не должен публиковаться наружу без дополнительного защищённого периметра;
- `Authorization` обязателен;
- `telemt-admin` использует точное значение `Authorization` из собственного `[telemt_api]`;
- bootstrap-скрипт должен генерировать безопасный `auth_header`;
- `/metrics` документируется отдельно и не рассматривается как замена control API;
- `rotate-secret` не используется до исправления в upstream `telemt`.

## Стратегия rollout

Rollout делится на три фазы:

1. Ввести backend-абстракцию, но оставить legacy-модель рабочей.
2. Включить API backend на одном сервере с `allow_file_fallback = true`.
3. После ручной проверки основных сценариев считать API-path основным.

## Проверяемые сценарии rollout

- регистрация через auto-approve токен;
- ручное одобрение pending-заявки;
- повторный запрос `/link` для существующего пользователя;
- удаление пользователя;
- экран `/service` и runtime health;
- деградация при API-сбое с переходом на fallback или с понятной ошибкой.

## Последствия

Положительные:

- control API становится безопасной и формализованной интеграционной точкой;
- ввод новых окружений стандартизируется;
- поведение при деградации заранее определено.

Негативные:

- добавляется операционная дисциплина вокруг `auth_header` и bind-конфигурации;
- требуется поддерживать и документировать rollout-сценарий;
- часть установок по-прежнему будет зависеть от legacy-прав на `telemt.toml` до полного отказа от fallback.

## Инварианты

- публичная экспозиция control API без auth считается неправильной конфигурацией;
- изменение `server.api` требует планового restart `telemt`;
- `/metrics` и control API обслуживают разные use-case'ы;
- rollout не считается завершённым, пока не проверены `/start`, approve, `/link`, delete и `/service`.
